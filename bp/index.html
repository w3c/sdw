Introduction

General introduction

To quote the [[DWBP]], there is a growing interest in publishing and consuming data on the Web. Very often the common factor across multiple datasets is the location data. Spatial data, or data related to a specific location, is what this Best Practice is all about.

There's already a lot of spatial data on the web. Everone knows the maps, satellite and street level images offered by search engines. There are many nice examples of spatial data being used in web applications.

SF Park is a website where users can look on a map of San Francisco to see where parking is available and at what price. Parking prices are incrementally raised or lowered in SFpark pilot areas based on demand. In this application, static data on existing parking spaces is combined with changing data about where people park the most, which is measured by parking sensors.

Different groups of people have a need for spatial data on the web.

Commercial operators, including search engines, invest a great deal of time and effort in generating geographical databases which are mirrors to Web content with the geographical context often added manually or at best semi-automatically. This process would be substantially aided if data were published on the Web with the appropriate geographic information at the source, thus allowing discovery and access using the standard mechanisms of the Web.

Geospatial experts who try to find and use data published in SDIs, can get frustrated by the fact that most web services available for spatial data are in fact WMS services, which serve static pictures, not data. WFS services, which allow you to get data, also exist, but are far less common. One could ask: do we really have a spatial data infrastructure or is it mostly just a 'spatial picture infrastructure'?

If you're not a geospatial expert, this may be the first you hear about SDI, WMS and WFS. But in fact, there is a whole world of geospatial standards, maintained by the OGC, aimed at publishing geospatial data in a related series of standardized web services and processing it with specialized tools.

The intended users of the SDI are experts in the geospatial domain. The OGC standards cover the full range of geospatial use cases – some of which are very complex. Because of this, it requires significant expertise in geospatial information technology to be able to use the SDI.

For web developers, who come from outside the geospatial domain, the data behind the OGC services is part of the “Deep Web” - the data is published behind specialized web services and not easy to get at, unless you're an expert. But web developers are increasingly creating and using data related to locations, so they are an important factor for geospatial data.

The public sector creates a lot of geospatial data, a lot of which is open data and could be useful to others, who may or may not be geospatial data experts. In addition, geospatial data is becoming more and more important for the web and its importance is still growing, among other things because of the rise of the internet of things. And this importance is far beyond serving static ‘pixels’. Semantically meaningful information on objects is required.

In short, there is a large demand for spatial data on the web. But there are some questions around publishing spatial data on the web that keep popping up and need to be answered. One is that many relevant standards already exist. These include informal 'community standards' - geospatial formats and/or vocabularies that enjoy widespread adoption (GeoJSON being the prime example) and others for which the formal standardization process has not been completed. Where standards have been completed there are competing ideas and engineers are often unsure which ones to adopt. With these factors in mind, this Best Practice aims to clarify and formalize the relevant standards landscape.

Questions answered in this document include how spatial information can best be integrated with other data on the Web; how machines and people can discover that different facts in different datasets relate to the same place, especially when 'place' is expressed in different ways and at different levels of granularity; how spatial aspects of metadata can be provided; and many other things.

Principles

We will not reinvent; the best practices in this document are based on what is already there. The added value is that they are assembled here.

The last thing we want is for this to become a 'best theories' document. This Best Practices document takes its examples and solutions from what is being done in practice right now. In other words, it is as much as possible evidence based. This document will also describe 'emerging practices' when no established solutions can be found in practice yet. Best versus emerging practices will be clearly recognizable in this document.

This document complies as much as possible with the principles of the [[LD-BP]] and the (developing) [[DWBP]]. Where it does not, this will be identified and explained.

Best practices have to be visible. This means: We will link to at least one (or two, three) publicly available example(s) of a non-toy dataset that follows the best practice.

There should be many (small) examples in the Best Practices, as examples can be a good way of communicating an idea and are a good way to start an implementation.

The Principles section is mainly written for the Spatial Data on the Web working group itself; this section will be removed from this document. Some of the text might go into other sections.

Audience

The introduction makes clear that we have multiple groups of potential users in mind with this document. One of these groups is the geospatial expert, and another is the web developer with no specific geospatial expertise. The first group can work with WFS; the second are better served with simpler interfaces in a more webby fashion. The audience is the broadest community of web users possible, application and tool builders addressing the needs to the mass consumer market should find value and guidance in the document.

Geospatial experts without web knowledge

They are vital in liberating existing spatial data, already published but not visible or linkable.

Web developers without geospatial knowledge

An aim of this document is making it easy for non-experts to work (find, publish, use) with spatial data on the web. Web application developers who just (!) want to use spatial data should not need to become GIS experts; spatial data is just one facet of the information space they work with, so let's not make it special by requiring up front knowledge.

Content publishers

The first two audience groups mentioned are user types; but spatial data publishers are a third important audience for this document. They want to know how to publish their spatial data in order for it to get used to its full potential.

Scope

This document is about spatial data on the web. Practices about publishing data in general on the web are out of scope. Sometimes they are mentioned in this document and general examples are given, together with references to relevant resources about publishing data on the web in general.

Spatial data can be data that has an explicit geometry as a property, but this is not necessarily the case. Data about a place is spatial data even if no explicit geometry can be given for this place.

It's out of scope for this document:

  • As already mentioned: to give best practices about publishing data in general on the web.
  • To give best practices for displaying spatial data on maps.
  • To discuss who has authority to mint URIs.
  • ...

Expressing spatial data

It is important to publish your spatial data with clear semantics. The primary use case for this is you already have a database of assets and you want to publish the semantics of this data. Another use case is someone wanting to publish some information wich has a spatial component on the web in a form that search engines will understand.

The spatial thing itself and its geometry property also has semantics. We may describe that here or in the Expressing geo section.

How should I publish my vocabulary? This question is answered in the [[LD-BP]] document. We recommend that you link your own vocabulary to commonly used existing ones because it increases its usefulness. How to do this is out of scope for this document; however, we give some examples of mapping relations you can use from OWL, SKOS, RDFS. And we do the mapping between some commonly used spatial vocabularies.

The current list of RDF vocabularies / OWL ontologies for spatial data being considered by the SDW WG are provided below. Some of these will be used in examples. Full details, including mapping between vocabularies, will be published in a complementary NOTE [tbd].

Vocabularies can discovered from Linked Open Vocabularies (LOV); using search terms or Tags place, Geography, Geometry and Time.

  • schema.org
  • OGC GeoSPARQL
  • neogeo
  • DBPedia (including dbpedia:Place)
  • UK Ordnance Survey geometry
  • UK Ordnance Survey spatial relations
  • UK Office for National Statistics http://statistics.data.gov.uk/def/statistical-geography# and http://statistics.data.gov.uk/def/statistical-entity# (URIs dont' resolve)
  • XKOS (used for geographical hierarchies in some examples)
  • Dublin core metadata 'DC Terms' (including dct:spatial and dct:coverage etc.)
  • BBC's Place
  • ISA Programme Location Core Vocabulary

  • VCard ontology includes definition of location

  • IGN ontology for describing geometry (re-using neogeo and geosparql)
  • IGN ontology for describing administrative units

Enabling discovery

This section is about using links for discovery, whereas is concerned with creating links.

SDIs only allow discovery at the dataset level but not at the data level (the spatial things and their properties), which is wat we want. Metadata on both the dataset and the data level are useful, and we recommend not to keep them separate (like in an SDI) but together.

The current OGC WxS services need some form of redesign (perhaps in the form of an extra layer on top aka a shim) in order to make it more discoverable and the data inside linkable. I.e. make crawlable pages to improve discovery.

The Data on the Web Best Practice is addressing discovery at the dataset level including spatial and temporal coverage.

GeoDCAT-AP is a spatial extension to DCAT which can help improve discovery of datasets. (see also ISSUE 35).

One way of discovering is browsing through data using semantics; which allows you to find related data through semantic similarity.

Linksets are useful for making links between entities in the dataset discoverable and crawlable. Using linksets are one example of doing that at the dataset level, a third party making collections of links browsable and allowing discovery of backlinks. Note that this is an emerging practice, not an established one. Also address here: how do you deal with maintaining links over time (when resources you link to may change or disappear). Example: OpenPHACTS.

This needs further discussion. Linksets does not seem to be an evidence-based best practice. It might be an emerging practice; CSIRO example. This might go in a shared note with the DWBP. An example of expressing a collection of links from the library community is here. An example vocabulary for expressing link sets is in VoID.

Datacubes can have a dimension which is a place name from a gazetteer and this allows the discovery of thematic data about a place.

Some data is behind a paywall or security because of privacy. Metadata for such datasets should be published in a way that it is crawlable even though the data itself is not within reach of the crawler. There are practices existing for this; the SDWBP will point to them.

Dataset-level discovery is covered by DWBP [ref?]; spatial data is typically more structured and contains Things that can be discovered individually

Exposing datasets through APIs

scope: API covers both the (query) interface used and the content exposed by the given endpoint

… 4 categories: question/answer, event based, pub/sub, transactional

using an API that supports content negotiation when necessary.

the URL for accessing a dataset does not equal a persistent identifier for the dataset itself.

There's no best practice for exposing Web Feature Service content in a more accessible way from a linked data point of view. However we describe here a recommended way of doing it, giving each feature its own stable URI and extending the API to make it more discoverable.

The current OGC WxS services need some form of redesign (perhaps in the form of an extra layer on top aka a shim) in order to make it more discoverable and the data inside linkable. To make the data inside a service more linkable stable URIs need to be minted for the spatial things in the service and be made available. This would require modifications to the service itself.
